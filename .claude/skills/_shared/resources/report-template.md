# 브리핑 템플릿 — 작업 완료 보고

백엔드 리더에게 보고하듯, **맥락 → 변경 → 영향 → 검증 → 차이 → 잔여** 순서로 구체적으로 브리핑한다.
"무엇을 했다"가 아니라 **"왜 이걸 했고, 어디까지 영향이 가고, 어떻게 확인했는지"**를 전달하는 것이 목적.

---

## 템플릿

```
## 브리핑

### 배경
{왜 이 작업이 필요했는지 — 요청 맥락 1-2줄}

### 변경 내역
| 파일 | 변경 내용 | 이유 |
|------|----------|------|
| `파일 경로` | 구체적 변경 | 왜 이 변경이 필요했는지 |

### 영향 범위
- **직접 영향**: {이 변경으로 동작이 바뀌는 부분}
- **간접 영향**: {연관되지만 변경 불필요한 부분}
- **영향 없음 확인**: {관련 있어 보이지만 무관한 부분}

### 검증
| 검증 항목 | 방법 | 결과 | 증거 |
|----------|------|------|------|
| {뭘 확인했는지} | {어떻게 확인했는지} | PASS/FAIL/미검증 | {실제 출력이나 근거} |

### 요청 대비 차이
- **추가**: {요청에 없었지만 추가한 것 + 이유}
- **미수정**: {요청 범위 밖이라 건드리지 않은 것}
- (차이 없으면 "요청 그대로 수행" 한 줄)

### 잔여 사항
- {남은 작업이나 사용자 확인 필요 사항} (없으면 생략)
```

---

## 작성 규칙

### 배경
- 사용자가 왜 이 작업을 요청했는지 맥락을 1-2줄로 요약
- 사용자 원문을 그대로 옮기지 않고, **작업 필요성**을 서술

### 변경 내역
- 파일 단위로 **뭘 바꿨고 왜 바꿨는지** 기술
- 파일 경로는 백틱으로 감싸기
- 변경 내용은 구체적으로 — "수정", "개선" 같은 추상어 금지
- 예: "수정" → "토큰 만료 시간 검증 로직 추가", "개선" → "에러 응답에 error_code 필드 포함"

### 영향 범위
- 3단 분류: 직접 / 간접 / 무관
- **직접 영향**: 이 변경으로 동작이 실제로 바뀌는 코드, API, 화면
- **간접 영향**: 의존성이 있지만 변경이 필요 없는 부분 (+ 왜 불필요한지 근거)
- **영향 없음 확인**: 관련 있어 보이지만 실제로 무관한 부분 (+ 무관한 이유)
- 확신이 없으면 "미확인" 표기 후 잔여 사항에 추가

### 검증
- 테이블 형식 필수: 항목 / 방법 / 결과 / 증거
- **방법**: 실행한 커맨드, 테스트 시나리오, 수동 확인 절차 등 구체적으로
- **결과**: `PASS` / `FAIL` / `미검증` 중 하나
- **증거**: 실제 출력, 로그, 스크린샷, 에러 메시지 발췌
- "확인했습니다"는 검증이 아님 — **어떻게 확인했고 뭐가 나왔는지**까지 기술
- 미검증 항목이 있으면 왜 못 했는지 + 누가 해야 하는지 명시

### 요청 대비 차이
- 사용자가 시킨 것과 실제 수행한 것의 차이를 투명하게 기술
- **추가**: 요청에 없었지만 추가한 것 (+ 추가한 이유)
- **미수정**: 요청 범위 밖이라 건드리지 않은 것 (+ 범위 밖 판단 근거)
- 차이 없으면 "요청 그대로 수행" 한 줄로 종결

### 잔여 사항
- 남은 작업, 사용자 확인 필요 사항, 후속 작업 제안
- 없으면 섹션 자체 생략

---

## 작업 규모별 적용

| 규모 | 적용 수준 |
|------|----------|
| 단순 (1-2파일, 명확한 변경) | 배경 + 변경 내역 + 검증. 나머지 생략 가능 |
| 보통 (3-5파일, 로직 변경) | 전체 템플릿 적용 |
| 복합 (다수 파일, 아키텍처 변경) | 전체 템플릿 + 영향 범위 상세화 |

---

## 예시: 단순 작업

```
## 브리핑

### 배경
로그인 API에서 만료된 토큰도 통과시키는 버그 수정 요청

### 변경 내역
| 파일 | 변경 내용 | 이유 |
|------|----------|------|
| `src/auth/service.ts:42` | `isTokenExpired()` 호출을 `validateToken()` 앞에 추가 | 만료 검증이 누락되어 있었음 |

### 검증
| 검증 항목 | 방법 | 결과 | 증거 |
|----------|------|------|------|
| 만료 토큰 차단 | 만료 JWT로 `POST /api/login` 호출 | PASS | `401 {"error": "token_expired"}` |
| 유효 토큰 통과 | 정상 JWT로 동일 API 호출 | PASS | `200` 정상 응답 |
| 기존 테스트 | `npm test -- auth` | PASS | 12/12 통과 |
```

## 예시: 복합 작업

```
## 브리핑

### 배경
인증 미들웨어를 세션 기반에서 JWT 기반으로 전환하는 작업

### 변경 내역
| 파일 | 변경 내용 | 이유 |
|------|----------|------|
| `src/auth/middleware.ts` | 세션 검증 → JWT 검증으로 교체 | JWT stateless 인증 전환 |
| `src/auth/service.ts` | `generateToken()`, `verifyToken()` 추가 | 토큰 생성/검증 로직 신규 |
| `src/config/auth.ts` | JWT secret, 만료 시간 설정 추가 | 환경별 토큰 설정 분리 |
| `src/types/auth.d.ts` | `TokenPayload` 타입 정의 | Request 객체 타입 확장 |

### 영향 범위
- **직접 영향**: 모든 인증 필요 API 엔드포인트 (15개) — 미들웨어 교체로 인증 흐름 변경
- **간접 영향**: 프론트엔드 로그인 로직 — 쿠키 → Authorization 헤더로 전환 필요
- **영향 없음 확인**: 공개 API (GET /health, GET /docs) — 인증 미들웨어 미적용 확인

### 검증
| 검증 항목 | 방법 | 결과 | 증거 |
|----------|------|------|------|
| 토큰 발급 | `POST /api/auth/login` 정상 자격증명 | PASS | JWT 정상 반환 |
| 토큰 인증 | 발급 토큰으로 `GET /api/users` 호출 | PASS | `200` + 사용자 목록 |
| 만료 토큰 | 1초 만료 토큰 생성 후 대기, 재요청 | PASS | `401 token_expired` |
| 위조 토큰 | 임의 문자열로 요청 | PASS | `401 invalid_token` |
| 기존 테스트 | `npm test` 전체 실행 | PASS | 47/47 통과 |
| 프론트 연동 | 미검증 | - | 프론트 배포 후 경수님 확인 필요 |

### 요청 대비 차이
- **추가**: `src/config/auth.ts` 환경별 설정 분리 — 하드코딩 대신 환경변수 사용이 운영에 안전
- **미수정**: 리프레시 토큰 — 요청 범위 밖, 별도 작업으로 분리 권장

### 잔여 사항
- 프론트엔드 Authorization 헤더 전환 필요
- 리프레시 토큰 구현은 별도 태스크로 제안
```
